Better Oblivion Sorting Software

A "one-click" program for users that quickly optimises and avoids detrimental conflicts in their TES IV: Oblivion, Nehrim - At Fate's Edge, TES V: Skyrim, Fallout 3 and Fallout: New Vegas mod load orders.

Features & Introduction
  • Correctly positions over 12,100 TES IV: Oblivion mods and counting, including such popular mods as FCOM, OOO, MMM, Better Cities, Open Cities, Race Balancing Project and many others.
  • Correctly positions over 2,600 Fallout 3 mods and counting.
  • Correctly positions over 2,500 Fallout: New Vegas mods and counting.
  • Provides support for sorting mods used in Nehrim - At Fate's Edge installs.
  • Provides preliminary support for TES V: Skyrim.
  • Provides a large number of usage notes for many mods, including requirements, incompatibilities, updates and bug warnings.
  • Provides detection and warning of installation mistakes for some popular mods, such as FCOM.
  • Displays the version number of mods when it is available.
  • Provides Bash Tag suggestions that are automatically applied by Wrye Bash (for Oblivion), Wrye Flash (for Fallout 3) or Wrye Flash NV (for Fallout: New Vegas).
  • Sorts mod files ghosted by Wrye Bash/Flash/Flash NV.
  • Allows easy user customisation of mod positions and the messages provided.
  • Includes a GUI for easily accessing all of Better Oblivion Sorting Software's functions, including launching it with different settings, and a semi-automatic updater to ensure you always have the latest version installed.
  • Is under constant development and improvement by a team of modders and experts on all supported games.

Better Oblivion Sorting Software (BOSS) will re-order your mods to their correct positions, putting any mods it doesn't recognise after them, in the same order as they were before BOSS was run.

BOSS is designed to assist in the installation and usage of mods, especially more complex mods such as FCOM, MMM, FOOK and nVamp, and help mod users avoid serious conflicts. It is not a complete solution to load ordering issues, as there are far more mods out there (estimated at about 25,000 for Oblivion, 12,000 for Fallout 3 and 6,000 for Fallout: New Vegas) than BOSS knows about. To properly place mods BOSS doesn't know about, a good working knowledge of mod load ordering for the relevant game is still necessary, for which some research and documentation reading will go a long way.



Installation & Basic Usage

Note that BOSS does not support Windows 2000 or earlier.

BOSS can be installed for use with TES IV: Oblivion, TES V: Skyrim, Fallout 3, Fallout: New Vegas or Nehrim - At Fate's Edge (a Total Conversion mod for Oblivion). Installation may either be automated, using the BOSS Installer package, or manually, using the archive package. Either method is simple, and instructions for manual installation are given below.

  1. If you have Wrye Bash, Wrye Flash or Wrye Flash NV, make sure the Lock Times function is deactivated.
  2. If upgrading from an earlier version of BOSS, delete all BOSS files and folders in your game's Data folder (including the BOSS folder, BOSS.exe, masterlist.txt, modlist.txt, modlist.old, BOSSlog.txt and any of BOSS's .bat files). If you have an edited masterlist, make sure to keep a backup so that your edits are not lost. It is recommended that you instead use the userlist functionality to apply changes to the masterlist.
  3. Extract the BOSS archive you downloaded to your game's installation directory (ie. the one containing the Data folder and the game's executable, so one of Oblivion, Fallout 3, Nehrim, fallout new vegas or skyrim depending on your game), so that there is now a BOSS folder alongside the Data folder and the game's executable.
  4. In the BOSS folder, run the BOSS.exe. This will download the latest masterlist from the Internet and then sort your mods.
  5. Thereafter when running BOSS, you may run it directly from the BOSS.exe to simply update your masterlist then sort your mods, or you can use the GUI to choose how BOSS runs.

If you are installing BOSS manually and have more than one of the supported games installed, you should install BOSS separately for each one.

Once BOSS has finished running, the output will be displayed automatically. Read any notes, errors if applicable, and make any relevant changes that are suggested by them. Wrye Bash/Flash/Flash NV will automatically import and apply Bash Tag suggestions from BOSS. Some Bash Tag suggestions can come with conditions and notes that require human intervention though, so read the suggestions makes thoroughly and adjust tags by hand in Wrye Bash/Flash/Flash NV if necessary.

You will need to reorder unrecognised mods by hand using OBMM/FOMM or Wrye Bash/Flash/Flash NV.



Uninstallation

Delete the BOSS folder in your game's installation folder (ie. the one containing the Data folder and the game's executable, so one of Oblivion, Fallout 3, Nehrim, fallout new vegas or skyrim depending on your game).



General & Advanced Usage

The Graphical User Interface

BOSS comes with a GUI that provides easy access to all of BOSS's functions and features. It allows you to:

  • Change BOSS's output options:
    • Show BOSSlog On Completion: Open the BOSSlog automatically after BOSS is finished.
    • Enable Debug Output: Add source code references to the command line output.
    • Log Command Line Output: Save the command line output to a BOSSCommandLineLog.txt text file.
    • BOSSlog Format: Specify the format of the BOSSlog generated, and the format that the GUI attempts to open when you click the View BOSSlog button.
    • Command Line Verbosity: The verbosity of the command line logging output. As values increase, so does verbosity, 'Standard (0)' displays only warning and error messages.
  • Easily select from the three main modes of operation BOSS can run in, and specify their run options:
    • Sort Mods: The standard mode of operation, sorting your mods. Its options are:
      • Update Masterlist: Run the masterlist updater during BOSS's operation.
      • Display Mod Versions: Display in the BOSSlog the version information extracted from plugin descriptions.
      • Display File CRCs: Calculate and display in the BOSSlog the CRCs of all the plugins in your Data folder, and of OBSE/FOSE/NVSE and its installed plugins.
      • Perform Trial Run: When BOSS is run, simulate its changes to your load order without actually reordering your mods.
    • Update Masterlist Only: Updates your masterlist then immediately quits. Its option is:
      • Game: The game is BOSS installed for. This affects which masterlist BOSS downloads and which master file to look for. 'Autodetect' tells BOSS to decide for itself based on what master file is present.
    • Undo Changes: Sorts your mods into the order they were in before you last ran BOSS, or the order they were in before the second-last time you ran BOSS, depending on the revert level chosen. Its options are:
      • Undo Level: Specify how many runs of BOSS to undo.
      • Display Mod Versions: Display in the BOSSlog the version information extracted from plugin descriptions.
      • Display File CRCs: Calculate and display in the BOSSlog the CRCs of all the plugins in your Data folder, and of OBSE/FOSE/NVSE and its installed plugins.
  • Change BOSS and GUI settings:
    • Check for BOSS updates on GUI startup: Controls whether or not the GUI checks for updates automatically when it starts up.
    • Proxy Type: Specify the type of proxy you use to connect to the Internet, if you use one.
    • Proxy Hostname: Specify the hostname of your proxy, if you use one.
    • Proxy Port Number: Specify the port number through which your proxy is accessed, if you use one.
  • Launch BOSS.
  • Open your userlist for editing.
  • Open your BOSSlog for viewing.
  • Open BOSS's ReadMes.
  • Launch the Check For Updates functionality, which includes an automatic updater that can get new releases of BOSS for you. The Check For Updates functionality also runs when the GUI launches, but only displays a message if an update was found or there was an error encountered while checking.

Each setting in the GUI corresponds to a command line parameter and/or an ini setting, the details of which can be found in the Further Details and Customising How BOSS Works sections of this ReadMe respectively.


The BOSS Log

The BOSSlog is where you will find all of BOSS's outputted information. It can be generated in either HTML format (the default) or as a plain text file with minimal formatting. The HTML format has significant advantages over the plain text format, being considerably easier to read, containing colour-coded messages and a variety of filters to customise what is displayed. Both formats, however, contain the same information.

The HTML BOSSlog has been tested in Mozilla Firefox 4, Google Chrome 11, Opera 11 and Internet Explorer 9. Using earlier browsers may result in parts of the BOSSlog not working or displaying correctly.

The BOSSlog comprises of a number of sections, them being:

  1. Masterlist Update: This section is displayed unless you have inhibited the masterlist updater via a GUI or BOSS.ini setting or the command line parameter, and details the success or failure of the update.
  2. General Messages: This section is only displayed if there are any applicable messages to be displayed. Its primary purpose is to allow BOSS to display messages specific to your Oblivion install that are not specific to any one mod.
  3. Summary: This section gives breakdowns of the numbers of plugins you have installed and the number of messages BOSS has attached to them. It also displays a notice if there has been no change to your recognised mod list since you last ran BOSS.
  4. Userlist Messages: This section is only displayed if BOSS finds a userlist.txt in your BOSS folder. In this section the successes or failures of the application of your userlist rules will be detailed.
  5. Script Extender And Script Extender Plugins Checksums: This section is displayed when BOSS is run with the Display File CRCs option enabled and you have your game's Script Extender installed. It will list the Script Extender and any plugins you have installed, displaying their versions if given and their CRCs.
  6. Recognised And Re-ordered Plugins or Restored Load Order: This section comprises the bulk of BOSS's output, listing all your recognised installed plugins and any messages BOSS has attached to them, as well as version information (if given) and the plugin CRCs (if run with Display File CRCs enabled). When BOSS is run to undo its changes, this section will list your restored load order.
  7. Unrecognised Plugins: This section lists any plugins you have installed that have not been recognised by BOSS. These plugins should be reported in an official BOSS thread with download links to their mods so they can be added to the masterlist.

BOSS will display the version numbers of any plugins that provide version information in their description fields. If a plugin does not supply any version information, no version number will be displayed for it. The version number of plugins may occasionally not match the version of the mod installed, this is due to the mod author forgetting to update the plugin's version information. When in doubt, check the version number given in a mod's readme.

It should be noted that the vast majority of BOSS's mod messages are non-conditional, in that they are not displayed in response to your specific install. For example, a message that gives some of the mod's incompatibilities does not know if you have them installed or not, and a message telling you to do something will not know if it has been done.

BOSS's mod messages are one of the most important features BOSS provides, acting as a means of providing users with information that they might otherwise not obtain. It is important for a stable, healthy game that you act on any messages that require action. If you think a message suggests an action that is not necessary, report it to an official BOSS thread.

The HTML BOSSlog includes a few features and a number of filters to customise what is displayed. The majority of these are found in the shaded box near the top of the BOSSlog, and are enabled and disabled via their checkboxes:

  • Collapsible Sections: This does not have a checkbox. Instead, clicking on a section's header will collapse that section, hiding its contents. Clicking the header again will expand the section, making its contents visible again.
  • Use Dark Colour Scheme: This will change the BOSSlog's colour scheme to a darker variety, more suitable for reading in low light levels and putting less strain on the eyes. Users with vision impairments may find the darker colour scheme easier to read.
  • Hide Rule Warnings: This hides any warning messages printed in the Userlist Messages section of the BOSSlog.
  • Hide Version Numbers: This will hide the version numbers printed in blue next to those mods that provide them.
  • Hide 'Ghosted' Label: This will hide the 'Ghosted' printed in grey next to any mods ghosted by Wrye Bash.
  • Hide Checksums: This will hide the checksums of mods and Script Extender plugins printed in brown next to their names.
  • Hide Messageless Mods: This will hide any recognised mods that do not have messages attached to them.
  • Hide Ghosted Mods: This will hide any mods that have been ghosted by Wrye Bash.
  • Hide Clean Mods: This will hide any mods in your load order that don't have messages beginning Contains dirty edits: attached. Note that this doesn't mean the mods hidden are definitely clean, only that BOSS doesn't know they're dirty.
  • Hide All Mod Nessages: This hides all the messages attached to all recognised mods.
  • Hide Notes: This hides any messages attached to mods that have the Note: prefix.
  • Hide Bash Tag Suggestions: This hides any Bash Tag suggestions attached to mods.
  • Hide Requirements: This hides any messages attached to mods that have the Requires: prefix.
  • Hide Incompatibilities: This hides any messages attached to mods that have the Incompatible with: prefix.

It is not possible to hide mod warning or error messages, as these are of a severity where they should almost certainly be acted upon, and not ignored. Be aware that if you hide all a mod's messages and then hide all messageless mods, that mod will be amongst those hidden.


Dirty Edits, Mod Cleaning & CRCs

Dirty edits are often a side-effect of mod creation, and are often due to bugs in the utilities Bethesda has provided to create mods, rather than bad practice by mod authors. Dirty edits can cause a wide range of issues, including incorrect game settings, missing content, broken quests and crashing to desktop. The more dirty edits there are, and the more mods with dirty edits you use, the more likely you are to experience issues.

Thankfully there is a way to remove dirty edits from mods relatively easily, a process known as mod cleaning, using TES4Edit (for Oblivion), FO3Edit (for Fallout 3) or FNVEdit (for Fallout: New Vegas). A detailed set of instructions and information on mod cleaning can be found on the CS Wiki's TES4Edit Cleaning Guide (the page also applies to Fallout 3 and Fallout: New Vegas).

The problem of dirty edits is largely a problem of ignorance on the part of mod authors and users alike of the problems unclean mods (more widely known as dirty mods) can cause. As such, there are community efforts to raise awareness of dirty edits and cleaning. BOSS plays a key role in these efforts, as it holds the complete list of all known dirty mods, and it uses this information to provide notification messages to users for any dirty mods they have installed. There are three key pieces of information that are used to accurately describe and identify an unclean mod. They are:

  • ITM Count: ITM (Identical To Master) records are a type of dirty edit where a mod has overwritten something in the game without actually changing anything. If another mod lower in the load order makes an intentional change to that thing, it will have its effect cancelled out by the ITM record, which can cause problems. The ITM count is the number of ITM records found and corrected in the mod plugin.
  • UDR Count: Deleting records is a significant cause of crashes and game bugs. UDR (Undeleted and Disabled Reference) records are records that the mod deleted from the game or from another mod that have been corrected safely by undeleting them and disabling them instead, which has the same intended effect without causing issues. The UDR count is the number of deleted reference records found and corrected in the mod plugin.
  • CRC or Checksum: A file's CRC is a highly accurate way of identifying it, and is a number calculated from the file's contents. If the contents of a file changes, so will its CRC. BOSS uses CRCs to check if a user's mods are dirty by checking their CRCs against pre-determined CRCs of known dirty mods. This lets BOSS give precise notification messages for any dirty mods it detects. BOSS can calculate the CRCs of all your installed plugins if run from the GUI with the Display File CRCs option enabled.

However, not all unclean mods have all or any of these pieces of information. For unclean mods whose ITM and UDR counts have not been reported, BOSS cannot give them. For unclean mods whose CRCs have not been reported, BOSS cannot give precise notification and so prints a message whether or not the user has cleaned their mod.

BOSS and the modding communities rely on user contribution of this information to progress. For information on how to contribute, see the Contributions & Help section below. If you find that a mod is dirty, you should also report this to the mod's author so that they can fix it.

There is also another type of unnecessary edit, known as a wild edit, that can also cause problems. These edits are neither Itentical To Master edits nor deleted records, so cannot be automatically cleaned. Cleaning must be manually done in TES4Edit/FO3Edit/FNVEdit. BOSS can still check for mods with these edits if the CRCs of the plugins with wild edits are known, and it can provide notification of these edits and some information on fixing them if they are reported in an official BOSS thread.


Updating Under Special Circumstances

The vast majority of users will be able to use BOSS's masterlist updater and the GUI's auto-updater with no problems. However, some users' circumstances may be such that the standard method will not work. This section provides instructions for those users.

The most common special circumstance is when the computer the user has BOSS installed on is not connected to the Internet. Merely transferring BOSS.exe to another computer and running it is not sufficient, as by default BOSS tries to autodetect the game it is installed for, and if there is no game present it will exit with an error. In this circumstance, the following instructions should be used to obtain masterlist updates:

  1. Copy BOSS.exe and BOSS GUI.exe from your BOSS folder to the computer with an Internet connection, ensuring they are in the same folder.
  2. If your Internet connection is through a proxy, you will also need to copy across your BOSS.ini to ensure the proxy settings you have recorded in it are applied. Make sure you have recorded your proxy settings in your BOSS.ini!
  3. Run the BOSS GUI.exe and select the Update Masterlist Only option. For the Game option below, select your game from the drop-down list. This will tell BOSS which game's masterlist to download.
  4. Select Run BOSS. Your masterlist will be downloaded into the same folder as your BOSS.exe and BOSS GUI.exe. A BOSSlog will also be created, you can delete this.

A rarer circumstance is when the user connects to the Internet using a proxy. This is most common in enterprise environments, and such users will generally know if they use a proxy. If you use a proxy to connect to the Internet, you must manually enter your proxy settings into your BOSS.ini settings file. See the section on the BOSS.ini in the Customising How BOSS Works section below for details. BOSS does not support the use of authenticated proxies at this time.

If your circumstances are such that the computer you have BOSS installed on is not connected to the Internet, or if you use a proxy to connect to the Internet, it is recommended that you avoid using the GUI's auto-updater, simply because it is easier in these circumstances to download new releases through your web browser.

There may also be other special cases which cause problems for BOSS's updaters that the BOSS team is unaware of. If you are having issues using the updaters, contact the BOSS team for support.


Customising How BOSS Works

Users can customise how BOSS handles their mods through the included User Rules feature. This feature allows you to define the positions of mods not in the masterlist, and to override the masterlist positions of mods, as well as edit the output messages of mods, and more. In addition, this is all done without editing the masterlist.txt, so when you update it using the built-in updater, your changes are not lost. All the information on the User Rules feature can be found in the "BOSS User Rules ReadMe.html" file.

For users who wish to be provided with a GUI feature for managing their userlist and own a copy of Microsoft Word 2003, 2007 or 2010, Surazal's BOSS Userlist Manager is a comprehensive utility that provides a range of features that may prove useful.

Users can also edit a number of BOSS's default settings in the BOSS.ini. As this is where BOSS gets its settings from, care must be taken to edit it correctly to prevent breaking BOSS's operation. The ini is split into the following sections:

  1. BOSS.InternetSettings: These settings control how the masterlist updater and GUI auto-updater connect to the Internet. They are shared by BOSS.exe and BOSS GUI.exe, and can be overriden for the masterlist updater using BOSS's command line parameters. They may also be edited through the GUI. You will probably know if you need to alter them.
    • ProxyType: The type of proxy used, if one is required to connect to the Internet. Valid values are: direct, http, http1_0, socks4, socks4a, socks5 and socks5h (SOCKS 5 with proxy-based hostname resolution). direct tells BOSS to use no proxy. Default is direct.
    • ProxyHostname: The hostname for the proxy host used, if a proxy is required to connect to the Internet. The hostname may be none, an actual hostname (eg. proxy.company.com) or an IP address (eg. 192.168.32.4). none tells BOSS to use no proxy. Default is none.
    • ProxyPort: The port used to connect to the proxy host, if a proxy is required to connect to the Internet. The port number may be any valid port number. 0 tells BOSS to use no proxy. Default is 0.
  2. BOSS.RunOptions: These settings specify BOSS.exe's run options. They allow you to run BOSS with non-default settings without having to use the command line or the GUI. Unless otherwise stated, they are binary toggles, so 1 switches them on and 0 switches them off.
    • UpdateMasterlist: Run the masterlist updater. Default is 1.
    • OnlyUpdateMasterlist: Exit BOSS immediately after running the masterlist updater. Default is 0.
    • DisableMasterlistUpdate: Inhibit the masterlist updater. Overrides UpdateMasterlist. Default is 0.
    • Game: The game is BOSS installed for. This affects which masterlist BOSS downloads and which master file to look for. Valid values are: auto, Oblivion, Nehrim, Fallout3, FalloutNV and Skyrim. auto tells BOSS to auto-detect the game. Default is auto.
    • SilentRun: Do not open the BOSSlog automatically after BOSS is finished. Default is 0.
    • NoVersionParse: Do not parse plugin header description fields for their version numbers. Default is 0.
    • RevertLevel: Specify how many runs of BOSS to undo. Valid values are: 0, 1 and 2. 0 tells BOSS to run as normal. Default is 0.
    • CommandLineVerbosity: The verbosity of the command line logging output. Valid values are: 0, 1, 2 and 3. As values increase, so does verbosity, 0 displays only warning and error messages. Default is 0.
    • Debug: Add source code references to debug output. Default is 0.
    • DisplayCRCs: Calculate and display the CRCs of all the plugins in your Data folder, and of OBSE/FOSE/NVSE and its installed plugins. Default is 0.
    • BOSSlogFormat: Specify the format of the BOSSlog generated. Valid values are: html, text. Default is html.
    • DoTrialRun: Run BOSS and simulate its changes to your load order without actually reordering your mods. Default is 0.
  3. GUI.Settings: These settings control aspects of the GUI's behaviour. As these settings are automatically loaded when the GUI starts and saved when it exits, there is no benefit to editing them by hand, so it is not recommended that you alter them manually.
  4. GUI.LastOptions: These settings record the options last selected by the user in the GUI. As these settings are automatically loaded when the GUI starts and saved when it exits, there is no benefit to editing them by hand, so it is not recommended that you alter them manually.
  5. BOSSlog.Filters: These settings control the default states of the BOSSlog filters. If a filter's default is set to be enabled, then it will be enabled and applied when the BOSSlog is opened. These settings are all binary toggles, so 1 enables them and 0 disables them. By default all filters are disabled. See the earlier The BOSS Log section for what each filter does.
    • UseDarkColourScheme: Controls the state of the Use Dark Colour Scheme filter.
    • HideVersionNumbers: Controls the state of the Hide Version Numbers filter.
    • HideGhostedLabel: Controls the state of the Hide 'Ghosted' Label filter.
    • HideChecksums: Controls the state of the Hide Checksums filter.
    • HideMessagelessMods: Controls the state of the Hide Messageless Mods filter.
    • HideGhostedMods: Controls the state of the Hide Ghosted Mods filter.
    • HideCleanMods: Controls the state of the Hide Clean Mods filter.
    • HideRuleWarnings: Controls the state of the Hide Rule Warnings filter.
    • HideAllModMessages: Controls the state of the Hide All Mod Messages filter.
    • HideNotes: Controls the state of the Hide Notes filter.
    • HideBashTagSuggestions: Controls the state of the Hide Bash Tag Suggestions filter.
    • HideRequirements: Controls the state of the Hide Requirements filter.
    • HideIncompatibilities: Controls the state of the Hide Incompatibilities filter.
  6. BOSSlog.Styles: These settings control the BOSSlog's CSS styles, used to provide the formatting for the BOSSlog. The defaults are not given, being hard-coded into BOSS, but they can be overridden here. A knowledge of CSS is expected if you wish to alter these settings. If you do not possess such knowledge, altering these settings is not recommended.

If BOSS is run and it cannot find a BOSS.ini, it will generate a new one. If BOSS is run with both a BOSS.ini present and command line parameters set, the command line parameters override the settings in BOSS.ini.


Running BOSS Under Linux

It is possible to run BOSS on Linux through the Wine compatibility layer, though no official support for doing so is given. Installation and usage is as on Windows. It is also possible to compile BOSS natively for Linux, and makefiles are provided, although there is again no official support for doing so. Native Linux builds are case-sensitive in their recognition of mods in the masterlist, and while mods are generally added with the correct case, occasionally a mod might have the incorrect case which would cause problems for a user with a native Linux build.



Further Details

Command Line Arguments

The BOSS GUI acts as a frontend to BOSS, and its output and run options correspond to BOSS's command line arguments. The syntax for these arguments is:

boss [-hVuosndctU] [-g [game]] [-r [level]] [-v [level]] [-f [type]] [-T [type]] [-H [hostname]] [-P [port]]

-h [ --help ]

This prints a short description of BOSS and some information on the function of the command line arguments, then exits.

-V [ --version ]

Prints the a banner displaying the program name and version number to the command line, then exits.

-u [ --update ]

Automatically updates the local copy of the masterlist using the latest version on the Google Code repository, and then sorts your mods. This behaviour is enabled by default, so the command line parameter will have no effect except when the automatic masterlist updating has been disabled in the ini.

-o [ --only-update ]

Automatically updates the local copy of the masterlist using the latest version on the Google Code repository, and then does not sort your mods.

-g [ --game ] [ game ]

Overrides game autodetection. The valid values of the argument are: Oblivion, Nehrim, Fallout3, FalloutNV and Skyrim. If the game specified is not detected, BOSS will fail before any mods are sorted, so this should be used in conjunction with -o.

-s [ --silent ]

Stops BOSS from automatically opening up the BOSSlog in your web browser (or text editor, depending on format chosen) after it has finished running.

-n [ --no-version-parse ]

Disables the parsing of each mod's description to extract the author's version information, so it is not printed alongside the other data in the BOSSlog. This can be used if the version parsing is not working properly for some of your mods and you wish to remove the version notes.

-r [ --revert ] [ level ]

Revert to a previous load order. This parameter optionally accepts values of 1 or 2, indicating how many undo steps to apply. If no option is specified, it defaults to 1. A value of 1 will restore the load order to that stored in modlist.txt. A value of 2 will restore the load order to that stored in modlist.old. If the modlist that you are reverting to does not exist, BOSS will cancel the undo and display an error message. The modlist files are not updated when BOSS is run with this parameter.

-v [ --verbose ] [ level ]

Specify verbosity level (0 to 3). 0 is the default, showing only warning and error messages. 1 (warning, error and informational messages) is implied if this option is specified without an argument. Higher values increase the verbosity further.

-d [ --debug ]

Adds source file references (filenames and line numbers) to logging statements.

-c [ --crc-display ]

Calculates the CRCs of all the plugins in your Data folder and displays them in your BOSSlog alongside the plugin names. The version numbers and CRCs of OBSE/FOSE/NVSE and its installed plugins will also be displayed in your BOSSlog.

-f [ --format ] [ type ]

Allows the selection of the BOSSlog format. Valid argument values are: html and text. html is the default output format. The text value will print the BOSSlog as plain text into a 'BOSSlog.txt' file, similar to the BOSSlog in BOSS v1.5 and below.

-t [ --trial-run ]

BOSS runs, but no mods are redated, so your load order is not changed. The BOSSlog still displays the order BOSS would have placed them in, so it is useful for checking what would happen if you ran BOSS on your load order without having to then undo BOSS's changes.

-U [ --no-update ]

Inhibits the default automatic masterlist updater, preventing it from running. Must not be used alongside --update,-u.

-T [ --proxy-type ] [ type ]

Sets the proxy type for the masterlist updater. Valid values are: direct, http, http1_0, socks4, socks4a, socks5 and socks5h (SOCKS 5 with proxy-based hostname resolution). direct is the default, and is equivalent to there being no proxy. Other types of proxy are not currently supported.

-H [ --proxy-host ] [ hostname ]

Sets the proxy hostname for the masterlist updater. The hostname may be an actual hostname (eg. proxy.company.com), an IP address (eg. 192.168.32.4) or none. none is the default, and is equivalent to there being no proxy.

-P [ --proxy-port ] [ port ]

Sets the proxy port number for the masterlist updater. The port number may be any valid port number. 0 is the default, and is equivalent to there being no proxy.


BOSS Changes and Files

The only changes BOSS makes to your game install are to re-date the .esp and .esm files in your Data folder, and create a few files in your BOSS folder. Your game's master file (Oblivion.esm, Fallout3.esm, Nehrim.esm or FalloutNV.esm) is never altered, and all dates are derived from it.

BOSS's files are all found in the BOSS folder, and they are:
  • BOSSlog.html - This log is where the output that BOSS generates and displays once the program ends is stored. It is replaced every time you run BOSS. You can view it in any web browser.
  • BOSSlog.txt - This file is the same as the above BOSSlog.html, but the information is given in plaintext instead of with HTML formatting. It is only created if you specify the 'text' format when running BOSS. You can view it in any text editor.
  • masterlist.txt - This is the master list of mods that BOSS recognises, listing them in order and which includes notes attached to them. It is the most frequently updated component of BOSS, receiving updates almost every day.
  • modlist.txt - This file contains the list of your mods in the order they were before you last ran BOSS, which can be used to undo changes to your load order.
  • modlist.old - This file contains the contents of your modlist.txt before you last ran BOSS, providing a second level of undo for load order changes.
  • userlist.txt - This file contains your custom rules to be implemented by the User Rules feature.
  • BOSS.ini - This file stores BOSS's settings, including the default run options, the default BOSSlog filters and more.
  • BOSS.ini.old - This file is a backup of your previous BOSS.ini, created when you update BOSS via the installer or the GUI's auto-updater.


Contributions & Help

General discussion, and the submission of questions, suggestions and unrecognised or unclean mods, takes place in BOSS's official threads for Oblivion, Nehrim, Fallout 3 and Fallout: New Vegas.

User contribution of additions and corrections to the masterlist are vital to the upkeep of BOSS. Let us know what you know.

Lists of unrecognised mods are welcome but the only way we can include them in the database in a timely way is if we get:

  • The ESP/ESM file name(s).
  • A link to where the description, mod or readme can be found to read or download.
  • A brief description of each mod will also help. Feel free to make load order suggestions.

Lists of unclean mods are also very welcome, but for the report to be most useful we require:

  • The ESP/ESM file name(s).
  • The number of records removed (the ITM count) as shown by TES4Edit/FO3Edit/FNVEdit (not the number of records processed).
  • The number of records undeleted (the UDR count) as shown by TES4Edit/FO3Edit/FNVEdit (not the number of records processed).
  • The CRC(s) of the unlcean ESP/ESM file name(s). Make sure that the CRCs are of the uncleaned files, not the files after cleaning. The CRC(s) may be calculated using BOSS, Wrye Bash/Flash/Flash NV or 7-zip, with other sources being unverified as correct. In the case of 7-zip, the CRC checksum for data is the one required.

Troubleshooting

Although BOSS is simple to use, it has grown to become a relatively complex application. As such, there are some circumstances under which BOSS may not be able to run correctly, due to a variety of possible reasons. In such cases, an error message detailing the problem should be provided. If the program unexpectedly crashes, or you experience a problem for which no error message is given, please report the crash or problem to an official BOSS thread (linked to in the Quick Links section below), so that we may fix it.

If the information below does not help you to solve any problems you are experiencing, please seek further support in one of the official BOSS threads.

General Problems

There are a few general reasons why BOSS may fail to function correctly. Check if any of the symptoms listed below fit your issue.

  • Symptom: No change in load order.
    • Issue: Wrye Bash's Lock Times load order locking feature is enabled.
    • Solution: Disable Lock Times before running BOSS. If you are using Wrye Bash v277 or later and running BOSS from Wrye Bash, this is done for you, and Lock Times is then re-activated after BOSS has run.
  • Symptom: BOSS does not change/update the BOSSlog and/or modlists correctly.
    • Issue: The BOSSlog and/or modlists are read-only, or the BOSS folder is read-only.
    • Solution: Ensure that the BOSS folder and any files in it are not read-only.
  • Symptom: BOSS does not recognise the mods in my userlist correctly, and/or replaces some of the letters in their names with question marks.
    • Issue: Your userlist is not encoded in UTF-8.
    • Solution: Save your userlist in the UTF-8 encoding.
    • Issue: You had mods in your userlist with non-Latin letters in their names, and saved your userlist in a non-UTF8 encoding before saving it in UTF-8.
    • Solution: Rewrite the filenames in your userlist, copy/pasting them from the files themselves to ensure they are correct.
  • Symptom: Some mods have the wrong or no version number given in the BOSSlog.
    • Issue: This occurs because BOSS looks for the version number in the plugin's description field. If the mod author has forgotten to update the version number there or not provided one, an incorrect or no version number will be displayed.
    • Solution: No solution, other than the mod author including a correct version number in the plugin's description field for a later release.

File Permissions

If you are running Windows Vista or Windows 7, BOSS (and other modding programs) may be prevented from working correctly by the UAC security feature. There are four common workarounds to this problem:

  1. Install the game outside the Program Files (or Program Files (x86) if on 64-bit Windows). UAC prevents unauthorised edits to these folders, so by installing the game outside of them, you remove it from UAC's reach, allowing you to use mods more easily.
  2. Deactivate UAC. This can be done from the Control Panel, and will turn UAC off across the whole of your computer. It is up to you whether you feel that you have adequate security measures in place to do this without risk.
  3. Give yourself Full Control permissions over your game install folder. This will allow you to make any edits you desire while keeping UAC active and on guard for any edits made by programs you do not run, and also means that you do not need to reinstall your game to a new location.
  4. Run BOSS as an Administrator. Right-click the BOSS.exe, and select Run as administrator.

For those that wish to take the third option and give themselves Full Control permissions, here is a guide:

  1. Right-click the folder you wish to change the permissions for, and select Properties.
  2. In Properties, select the Security tab, and click the Edit... button. A UAC prompt may appear, simply allow yourself to continue.
  3. In the Group or user names box, select the Users option.
  4. In the lower box, check the box opposite Full Control in the Allow column. Press OK to exit. If a UAC prompt appears, allow the change.
  5. In the Properties window, select OK to exit. You should now have Full Control permissions over your chosen folder.

International Character Support

BOSS provides support for sorting mods containing characters from any language via the Unicode character encoding, specifically UTF-8. All BOSS's files are encoded in UTF-8 by default. However, if you created your userlist.txt before BOSS automatically created one, you will probably need to convert it to the UTF-8 encoding, as it is not Notepad's default encoding. Below are some instructions for converting a file's encoding to UTF-8:

  • Using Notepad:
    1. Open the file in Notepad.
    2. Open the File menu, then select Save As....
    3. In the Save As dialog window that opens, there is a drop-down list labelled Encoding:. Select UTF-8 from this list. Do not select any of the other Unicode encodings, they will not work.
    4. Click the Save button, answering Yes to any confirmation prompts.
  • Using Notepad++:
    1. Open the file in Notepad++.
    2. Open the Encoding menu, then select Convert to UTF-8.
    3. Save the file as normal.

A file only needs to be converted to the correct encoding once, thereafter it will continue to save in the correct encoding whenever you edit it.
