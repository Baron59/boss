Features & Introduction

  • Correctly positions over 19,100 mod plugins and counting for TES IV: Oblivion, Fallout 3, Fallout: New Vegas and Nehrim - At Fate's Edge.
    • Over 13,000 / 2,900 / 3,100 / 180 mod plugins respectively.
  • Supports mod plugins for TES V: Skyrim.
  • Provides a large number of usage notes for many plugins, including requirements, incompatibilities, updates and bug warnings.
  • Provides detection and warning of installation mistakes for some popular mods, such as FCOM.
  • Displays the version number of plugins when it is available.
  • Provides Bash Tag suggestions that are automatically applied by Wrye Bash (for Oblivion), Wrye Flash (for Fallout 3), Wrye Flash NV (for Fallout: New Vegas) or Wrye Smash (for Skyrim).
  • Sorts plugins ghosted by the Wrye *ash utilities.
  • Allows easy user customisation of mod positions and the messages provided.
  • Includes both CLI and GUI clients, offering full functionality and ease-of-use for all users.
  • Includes a semi-automatic updater to ensure that you always have the latest version installed.
  • Is under constant development and improvement by a team of modders and experts on all supported games.

Better Oblivion Sorting Software (BOSS) will re-order your mod plugins to their correct positions, putting any mods it doesn't recognise after them, in the same order as they were before BOSS was run.

BOSS is designed to assist in the installation and usage of mods, especially more complex mods such as FCOM, MMM, FOOK and nVamp, and help mod users avoid serious conflicts. It is not a complete solution to load ordering issues, as there are far more mods out there (estimated at about 27,000 for Oblivion, 12,000 for Fallout 3 and 6,000 for Fallout: New Vegas) than BOSS knows about. To properly place mods BOSS doesn't know about, a good working knowledge of mod load ordering for the relevant game is still necessary, for which some research and documentation reading will go a long way.

Installation & Basic Usage

Note that BOSS does not support Windows 2000 or earlier.

BOSS can be installed for use with TES IV: Oblivion, TES V: Skyrim, Fallout 3, Fallout: New Vegas or Nehrim - At Fate's Edge (a Total Conversion mod for Oblivion). Installation may either be automated, using the BOSS Installer package, or manually, using the archive package. Either method is simple, and instructions for manual installation are given below.

  1. If you have any of the Wrye *ash utilities, make sure the Lock Times function is deactivated.
  2. If upgrading from an earlier version of BOSS, delete all BOSS files and folders in your game's Data folder (including the BOSS folder, BOSS.exe, masterlist.txt, modlist.txt, modlist.old, BOSSlog.txt and any of BOSS's .bat files). If you have an edited masterlist, make sure to keep a backup so that your edits are not lost. It is recommended that you instead use the user rules functionality to apply changes to the masterlist.
  3. Extract the BOSS archive you downloaded to your game's installation directory (ie. the one containing the Data folder and the game's executable, so one of Oblivion, Fallout 3, Nehrim, fallout new vegas or skyrim depending on your game), so that there is now a BOSS folder alongside the Data folder and the game's executable.
  4. In the BOSS folder, run the BOSS.exe. This will download the latest masterlist from the Internet and then sort your mods.

If you are installing BOSS manually and have more than one of the supported games installed, you should install BOSS separately for each one.

Once BOSS has finished running, the output will be displayed automatically. Read any notes, errors if applicable, and make any relevant changes that are suggested by them. The Wrye *ash utilities will automatically import and apply Bash Tag suggestions from BOSS. Some Bash Tag suggestions can come with conditions and notes that require human intervention though, so read the suggestions makes thoroughly and adjust tags by hand in Wrye *ash if necessary.

You will need to reorder unrecognised mods by hand using OBMM/FOMM or Wrye *ash.

Uninstallation

If you installed BOSS using the installer, then use the uninstaller it provides a link to in your Start Menu to uninstall BOSS. If you installed BOSS manually, do the following:

  1. Delete the BOSS folder in your game's installation folder (ie. the one containing the Data folder and the game's executable, so one of Oblivion, Fallout 3, Nehrim, fallout new vegas or skyrim depending on your game).

General Usage

This section gives more information on the parts of BOSS's functionality that users are likely to come into contact with during the course of general usage of BOSS. It gives details on the capabilities of the GUI, information on how to keep BOSS up to date, information on the BOSS Log and its constituent parts, and provides some background information and BOSS-specific notes on the subject of dirty edits and mod cleaning.

The Graphical User Interface

While the CLI provided by BOSS.exe offers the simplest way to run BOSS using just one click, the GUI provided by BOSS GUI.exe provides a way of accessing BOSS's options, settings and more powerful features without having to edit any other files manually.

The CLI and GUI both possess the same core functionality, but differ in how they approach providing it, with the CLI being text-based and the GUI using graphical elements such as buttons, checkboxes and drop-down lists. The GUI is composed of three windows: the Main window, which is the one displayed when the GUI is run; the Settings window; and the User Rules Editor window. Their content shall be explained in turn.

The Main Window

The Main window is where BOSS's most commonly-used options are located, along with buttons for running BOSS, editing your User Rules and opening your BOSS Log. The window's menu bar also includes additional options for accessing the Settings window, opening BOSS's ReadMe files, checking for updates and more. The options in the Main window are:

  • Output Options:
    • Show BOSS Log On Completion: Open the BOSS Log automatically after BOSS is finished.
    • Display Plugin Versions: Display in the BOSS Log the version information extracted from plugin descriptions where available.
    • Display File CRCs: Calculate and display in the BOSS Log the CRCs of all the plugins in your Data folder, and of OBSE/FOSE/NVSE and its installed plugins if present.
    • BOSS Log Format: Specify the format of the BOSS Log generated, and the format that the GUI attempts to open when you click the View BOSS Log button.
  • Run Options:
    • Sort Plugins: The standard mode of operation, sorting your plugins. Its options are:
      • Update Masterlist: Run the masterlist updater during BOSS's operation.
      • Perform Trial Run: Simulate BOSS's changes to your load order without actually reordering your mods.
    • Update Masterlist Only: Updates your masterlist then immediately quits. Its option is:
      • Game: The game is BOSS installed for. This affects which masterlist BOSS downloads and which master file to look for. 'Autodetect' tells BOSS to decide for itself based on what master file is present.
    • Undo Changes: Sorts your mods into the order they were in before you last ran BOSS, or the order they were in before the second-last time you ran BOSS, depending on the revert level chosen. Its option is:
      • Undo Level: Specify how many runs of BOSS are to be undone.
  • Buttons:
    • Edit User Rules: If you have the Use User Rules Editor setting in the Settings window enabled, clicking this will open the User Rules Editor window. Otherwise, it will open your userlist.txt in your default text editor.
    • Run BOSS: Runs BOSS's primary functionality with the options you have selected.
    • View BOSS Log: Opens your BOSS Log in the format selected by the BOSS Log Format option.

The Settings Window

The Settings window is where the rest of BOSS's options are located. These options are grouped into tabs depending on what area of BOSS's behaviour they control. The settings are:

  • General settings:
    • Check for BOSS updates on startup: Controls whether or not BOSS checks for updates automatically when it starts up.
    • Use User Rules Editor: Controls whether the GUI uses the User Rules Editor window to edit user rules or whether it opens the userlist.txt for manual editing.
  • Debugging settings:
    • Debug Outout Verbosity: The verbosity of the debug output. As values increase, so does verbosity, 'Standard (0)' displays only warning and error messages.
    • Include Source Code References: Add source code references to the debug output.
    • Log Debug Output: Save the debug output to a BOSSDebugLog.txt text file.
  • Internet settings:
    • Proxy Hostname: Specify the hostname of your proxy, if you use one.
    • Proxy Port Number: Specify the port number through which your proxy is accessed, if you use one.
  • BOSS Log Filters settings:
    • These settings control the default states of the BOSS Log filters. If a setting's box is checked, then the filter of the same name in the BOSS Log will be enabled when you open the BOSS Log. See the The BOSS Log section below for what each filter does.
  • BOSS Log Styling settings:
    • These settings control the BOSS Log's CSS styles, used to provide the formatting for the BOSS Log. A knowledge of CSS is expected if you wish to alter these settings. If you do not possess such knowledge, altering these settings is not recommended.
    • The CSS selectors are given on the left, and their styling can be edited in the corresponding text boxes on the right.

Each setting in the GUI corresponds to a command line parameter and/or an ini setting, the details of which can be found in the Command Line Arguments and Customising How BOSS Works sections of this ReadMe respectively.

The User Rules Editor

The User Rules Editor can be used to manage your user rules using a graphical interface instead of editing your userlist.txt directly. The information contained within the BOSS User Rules ReadMe.html document is still required to understand user rules. The editor contains the following:

  • User Rules:
    • This section lists your user rules in order.
    • Rules that are currently enabled have their checkboxes checked, whereas disabled rules are unchecked.
    • Rules can be moved up or down using the Move Rule Up and Move Rule Down buttons at the bottom of the window.
    • When the User Rules Editor is loaded, it will automatically disable any rules adding mods that are already in the masterlist.
  • Installed Mods:
    • This section lists the plugins found in your Data folder. Plugins are listed in the order that they are loaded.
    • All plugins installed are displayed, not just those in your currently active load order.
    • The installed mod list may be searched using the search box at the top. Searching looks for matches from the start of plugin filenames, so searching for "xul" would return all plugins with filenames that start with "xul". Searching starts when the "Enter" keyboard key is pressed, it is not performed as you type.
    • A plugin may be selected in the list and drag 'n' dropped into a text box in the Rule Creator/Editor to add that plugin to the text box, rather than having to write out the plugin filename by hand.
  • Masterlist:
    • This section displays the masterlist as a tree list, showing plugins and groups.
    • The tree list is by default fully collapsed. Double clicking on a collapsed group (any row without a ".esp" or ".esm" in its name) will expand that group, showing its children. Double clicking an expanded group will collapse it again, hiding its children.
    • Selecting a plugin will cause the messages attached to it by the masterlist to be displayed in the Masterlist Mod Messages section.
    • The masterlistlist may be searched using the search box at the top. Searching looks for matches from the start of plugin filenames and group names, so searching for "xul" would return all plugins with filenames that start with "xul" and groups with names that start with "xul". Searching starts when the Enter keyboard key is pressed, it is not performed as you type.
  • Masterlist Mod Messages:
    • This section displays the messages attached to the currently selected mod in the Masterlist section. The messages are displayed as they appear in the masterlist, not as they appear in the BOSS Log. See the BOSS User Rules ReadMe.html for a brief overview of the most common message syntax, and the BOSS Masterlist Syntax.html documentation for the full details.
    • The messages cannot be edited here: they must be replaced or added to via a user rule in the Rule Creator.
  • Rule Creator/Editor:
    • This section is where the user edits or creates user rules. Selecting a rule in the User Rules list will load it into this section.
    • The rule object is specified in the first text box. This is the plugin or group that the rule acts upon.
    • If the rule object is to be sorted, the object it is sorted relative to must be specified in one of the two text boxes in the Sort Item subsection, depending whether the user is sorting the object relative to another, or inserting a mod into a group.
    • Messages for addition or to replace existing messages should be written in the large text box. They must use valid syntax.
    • To add the current contents of the Rule Creator/Editor as a new rule, use the Create New Rule button. To save edits to the selected rule in the User Rules list, use the Save Edited Rule button. The User Rules Editor will ask for confirmation before applying the edits.
    • To delete the rule selected in the User Rules list, use the Delete Rule button. The User Rules Editor will ask for confirmation before deleting the rule.
    • The User Rules Editor will catch most, if not all, syntax errors in created and edited rules and warn the user of the errors encountered. No invalid rules will be saved.

To save your changes to your userlist and exit, use the Save button. To exit without saving, use the Cancel button.

Plugins and groups may be added to the text boxes in the Rule Creator by dragging and dropping them from the Installed Mods and Masterlist lists on the right of the window. Dropping a plugin or group into a text box currently containing a plugin or group will cause the existing text to be replaced.

The User Rules Editor ignores all silent comments in your userlist.txt, and will strip them from the file if you choose to save your user rules via the editor.

Keeping BOSS Up-To-Date

It is important that you keep your BOSS installation updated, not only to benefit from the many frequent updates made to the masterlists, but also to benefit from bug fixes, improvements and new features introduced to new releases of BOSS itself.

Providing there is an Internet connection available when BOSS runs, it will automatically download and apply any updates available for the masterlists. BOSS will also check for new releases and if an update is available, it will ask the user if they want to download and install it. The check for new releases functionality behaves as outlined below:

  1. On launching, it automatically checks for updates to BOSS. If an update is found, it proceeds to the next step. If no update is available, or no Internet connection is present, it will display no message.
    • If you run the Check For Updates option from the GUI's Help option, then it will display messages detailing if no update was found or if there were any errors.
  2. No matter how the update check was started, if an update is available it will prompt the user whether or not to download and install the update. If the user decides not to update, the updater will quit.
  3. If the user decides to update, then it will begin downloading the file(s) to the BOSS folder, displaying the progress. If using the GUI, the download can be cancelled at any time, in which case any downloaded files will be deleted and the updater will quit. CLI users can cancel the download by quitting BOSS, but no cleanup will take place.
    • If you previously installed BOSS using the installer, then the updater shall download the installer for the update, otherwise it will download the files as they appear in the manual archive.
  4. Once the download is complete, the updater will install the downloaded file(s).
    • If the installer was downloaded, it will be launched, and BOSS shall exit. Complete the installation wizard to finish updating.
    • If the manual archive files were downloaded, the old BOSS files will be deleted and replaced with the new ones. The updater will be unable to replace the executable that you are running BOSS from, but will instead tell you to rename the downloaded executable, then exit. You must manually rename the file, deleting the old executable, to complete the update.

Under certain circumstances the masterlist and program updaters may be unable to function fully. If you have issues using them, see the Updating Under Special Circumstances section for alternative methods.

BOSS program updates are also published at the TES Nexus, Skyrim Nexus, Fallout 3 Nexus and New Vegas Nexus BOSS pages. For those not using the masterlist updater, the latest masterlist is always on the Google Code repository.

Always make sure to use the latest version of BOSS and the latest masterlist.


The BOSS Log

The BOSS Log is where you will find all of BOSS's outputted information. It can be generated in either HTML format (the default) or as a plain text file with minimal formatting. The HTML format has significant advantages over the plain text format, being considerably easier to read, containing colour-coded messages and a variety of filters to customise what is displayed. However, other than the HTML BOSS Log providing a notification if there has been no change in the recognised mod list since last run, there is no difference in information provided between the two formats.

The HTML BOSS Log has been tested in Mozilla Firefox 4, Google Chrome 11, Opera 11 and Internet Explorer 9. Using earlier browsers may result in parts of the BOSS Log not working or displaying correctly.

The BOSS Log comprises of a number of sections, which are detailed in the table below.

SummaryThis section gives a short summary of BOSS's actions. It breaks down the numbers of plugins installed and messages attached to them and gives the local masterlist version and whether or not it was updated. If the BOSS Log is in the HTML format it will also display a message if your recognised plugin list has not changed since you last ran BOSS.

BOSS will display the version numbers of any plugins that provide them in their description fields. If a plugin supplies no version number, it cannot be displayed. The version numbers may not reflect the actual version of the mod installed - this could either be due to the mod author forgetting to update the description, or due to a mistake in BOSS's detection. As version numbers are supplied by mod authors in a wide range of formats and BOSS has to detect as many as possible, it occasionally identifies parts of some descriptions as version numbers incorrectly. This is largely unavoidable, but any discrepancies should be reported to the BOSS team in case they can be fixed. When in doubt, check the version number given in a mod's readme.

It should be noted that the vast majority of BOSS's mod messages are non-conditional, in that they are not displayed in response to your specific install. For example, a message that gives some of the mod's incompatibilities does not know if you have them installed or not, and a message telling you to do something will not know if it has been done.

BOSS's mod messages are one of the most important features BOSS provides, acting as a means of providing users with information that they might otherwise not obtain. It is important for a stable, healthy game that you act on any messages that require action. If you think a message suggests an action that is not necessary, report it to an official BOSS thread.

Dirty Edits, Mod Cleaning & CRCs

Dirty edits are often a side-effect of mod creation, and are often due to bugs in the utilities Bethesda has provided to create mods, rather than bad practice by mod authors. Dirty edits can cause a wide range of issues, including incorrect game settings, missing content, broken quests and crashing to desktop. The more dirty edits there are, and the more mods with dirty edits you use, the more likely you are to experience issues.

Thankfully there is a way to remove dirty edits from mods relatively easily, a process known as mod cleaning, using TES4Edit (for Oblivion), FO3Edit (for Fallout 3) or FNVEdit (for Fallout: New Vegas). A detailed set of instructions and information on mod cleaning can be found on the CS Wiki's TES4Edit Cleaning Guide (the page also applies to Fallout 3 and Fallout: New Vegas). Wrye *ash can also scan plugins for dirty edits and return their ITM and UDR counts, though at the time of writing it cannot clean plugins itself.

The problem of dirty edits is largely a problem of ignorance on the part of mod authors and users alike of the problems unclean mods (more widely known as dirty mods) can cause. As such, there are community efforts to raise awareness of dirty edits and cleaning. BOSS plays a key role in these efforts, as it holds the complete list of all known dirty mods, and it uses this information to provide notification messages to users for any dirty mods they have installed. There are three key pieces of information that are used to accurately describe and identify an unclean mod. They are:

  • ITM Count: ITM (Identical To Master) records are a type of dirty edit where a mod has overwritten something in the game without actually changing anything. If another mod lower in the load order makes an intentional change to that thing, it will have its effect cancelled out by the ITM record, which can cause problems. The ITM count is the number of ITM records found and corrected in the mod plugin.
  • UDR Count: Deleting records is a significant cause of crashes and game bugs. UDR (Undeleted and Disabled Reference) records are records that the mod deleted from the game or from another mod that have been corrected safely by undeleting them and disabling them instead, which has the same intended effect without causing issues. The UDR count is the number of deleted reference records found and corrected in the mod plugin.
  • CRC or Checksum: A file's CRC is a highly accurate way of identifying it, and is a number calculated from the file's contents. If the contents of a file changes, so will its CRC. BOSS uses CRCs to check if a user's mods are dirty by checking their CRCs against pre-determined CRCs of known dirty mods. This lets BOSS give precise notification messages for any dirty mods it detects. BOSS can calculate the CRCs of all your installed plugins if run from the GUI with the Display File CRCs option enabled.

However, not all unclean mods have all or any of these pieces of information. For unclean mods whose ITM and UDR counts have not been reported, BOSS cannot give them. For unclean mods whose CRCs have not been reported, BOSS cannot give precise notification and so prints a message whether or not the user has cleaned their mod.

BOSS and the modding communities rely on user contribution of this information to progress. For information on how to contribute, see the Contributions & Help section below. If you find that a mod is dirty, you should also report this to the mod's author so that they can fix it.

Note: TES4Edit will include a variety of junk records in the ITM count for a plugin, such as new empty cells that are automatically generated by the Construction Set and are almost impossible to remove. These junk records are non-harmful, and BOSS will inform users when a mod contains these to avoid confusion. This is also the source of any discrepancies between the counts produced by TES4Edit and those produced by Wrye *ash. The latter's counts are more technically accurate, but both are acceptable for use.

There is also another type of unnecessary edit, known as a wild edit, that can also cause problems. These edits are neither Itentical To Master edits nor deleted records, so cannot be automatically cleaned. Cleaning must be manually done in TES4Edit/FO3Edit/FNVEdit. BOSS can still check for plugins with these edits if the CRCs of the plugins with wild edits are known, and it can provide notification of these edits and some information on fixing them if they are reported in an official BOSS thread.

Contributions & Help

General discussion, and the submission of questions, suggestions and unrecognised or unclean plugins, takes place in BOSS's official threads for Oblivion, Nehrim, Skyrim, Fallout 3 and Fallout: New Vegas.

User contribution of additions and corrections to the masterlist are vital to the upkeep of BOSS. Let us know what you know.

Lists of unrecognised plugins are welcome but the only way we can include them in the database in a timely way is if we get:

  • The ESP/ESM file name(s).
  • A link to where the description, mod or readme can be found to read or download.
  • A brief description of each mod will also help. Feel free to make load order suggestions.

Lists of unclean plugins are also very welcome, but for the report to be most useful we require:

  • The ESP/ESM file name(s).
  • The number of records removed (the ITM count) as shown by TES4Edit/FO3Edit/FNVEdit or Wrye *ash (not the number of records processed).
  • The number of records undeleted (the UDR count) as shown by TES4Edit/FO3Edit/FNVEdit or Wrye *ash (not the number of records processed).
  • The CRC(s) of the unlcean ESP/ESM file name(s). Make sure that the CRCs are of the uncleaned files, not the files after cleaning. The CRC(s) may be calculated using BOSS, Wrye *ash or 7-zip, with other sources being unverified as correct. In the case of 7-zip, the CRC checksum for data is the one required.

Advanced Usage

This section gives information on how to operate BOSS under certain conditions, how to customise its behaviour more fully than previously covered, and provides details for all the BOSS files.

User Rules

Users can customise how BOSS handles their plugins through the included User Rules feature. This feature allows you to define the positions of plugins not in the masterlist, and to override the masterlist positions of plugins, as well as edit the output messages of plugins, and more. In addition, this is all done without editing the masterlist.txt, so when you update it using the built-in updater, your changes are not lost. All the information on the User Rules feature can be found in the "BOSS User Rules ReadMe.html" file that accompanies this ReadMe.

As this is where BOSS gets its settings from when both running the CLI client and when loading the GUI client, care must be taken to edit it correctly to prevent breaking BOSS's operation. If BOSS is run and it cannot find a BOSS.ini, it will generate a new one with default settings. If BOSS is run with both a BOSS.ini present and command line parameters set, the command line parameters override the settings in BOSS.ini. The ini is split into the following sections:

Command Line Arguments

While the BOSS GUI allows you to specify options using its graphical elements, the CLI uses a combination of the BOSS.ini file and command line arguments to do the same. The syntax for these arguments is:

boss [-hVuosndctUl] [-g [game]] [-r [level]] [-v [level]] [-f [type]] [-H [hostname]] [-P [port]]

-h [ --help ]

This prints a short description of BOSS and some information on the function of the command line arguments, then exits.

-V [ --version ]

Prints the a banner displaying the program name and version number to the command line, then exits.

-u [ --update ]

Automatically updates the local copy of the masterlist using the latest version on the Google Code repository, and then sorts your plugins. This behaviour is enabled by default, so the command line parameter will have no effect except when the automatic masterlist updating has been disabled in the ini.

-U [ --no-update ]

Inhibits the default automatic masterlist updater, preventing it from running. Must not be used alongside --update,-u.

-o [ --only-update ]

Automatically updates the local copy of the masterlist using the latest version on the Google Code repository, and then does not sort your plugins.

-g [ --game ] [ game ]

Overrides game autodetection. The valid values of the argument are: Oblivion, Nehrim, Fallout3, FalloutNV and Skyrim. If the game specified is not detected, BOSS will fail before any plugins are sorted, so this should be used in conjunction with -o.

-s [ --silent ]

Stops BOSS from automatically opening up the BOSS Log in your web browser (or text editor, depending on format chosen) after it has finished running.

-c [ --crc-display ]

Calculates the CRCs of all the plugins in your Data folder and displays them in your BOSS Log alongside the plugin names. The version numbers and CRCs of OBSE/FOSE/NVSE and its installed plugins will also be displayed in your BOSS Log.

-n [ --no-version-parse ]

Disables the parsing of each plugin's description to extract the author's version information, so it is not printed alongside the other data in the BOSS Log. This can be used if the version parsing is not working properly for some of your plugins and you wish to remove the version notes.

-f [ --format ] [ type ]

Allows the selection of the BOSS Log format. Valid argument values are: html and text. html is the default output format. The text value will print the BOSS Log as plain text into a 'BOSSlog.txt' file, similar to the BOSS Log in BOSS v1.5 and below.

-t [ --trial-run ]

BOSS runs, but no plugins are redated, so your load order is not changed. The BOSS Log still displays the order BOSS would have placed them in, so it is useful for checking what would happen if you ran BOSS on your load order without having to then undo BOSS's changes.

-r [ --revert ] [ level ]

Revert to a previous load order. This parameter optionally accepts values of 1 or 2, indicating how many undo steps to apply. If no option is specified, it defaults to 1. A value of 1 will restore the load order to that stored in modlist.txt. A value of 2 will restore the load order to that stored in modlist.old. If the modlist that you are reverting to does not exist, BOSS will cancel the undo and display an error message. The modlist files are not updated when BOSS is run with this parameter.

-v [ --verbose ] [ level ]

Specify the verbosity level (0 to 3) of the debugging output. 0 is the default, showing only warning and error messages. 1 (warning, error and informational messages) is implied if this option is specified without an argument. Higher values increase the verbosity further.

-d [ --debug-with-source ]

Adds source file references (filenames and line numbers) to debugging statements.

-l [ log-debug ]

Logs the debug output to the BOSSDebugLog.txt file instead of the command line.

-H [ --proxy-host ] [ hostname ]

Sets the proxy hostname for the masterlist updater. The hostname may be an actual hostname (eg. proxy.company.com), an IP address (eg. 192.168.32.4) or none. none is the default, and is equivalent to there being no proxy.

-P [ --proxy-port ] [ port ]

Sets the proxy port number for the masterlist updater. The port number may be any valid port number. 0 is the default, and is equivalent to there being no proxy.


Updating Under Special Circumstances

The vast majority of users will be able to use BOSS's masterlist updater and the auto-updater with no problems. However, some users' circumstances may be such that the standard method will not work. This section provides instructions for those users.

The most common special circumstance is when the computer the user has BOSS installed on is not connected to the Internet. Merely transferring BOSS.exe to another computer and running it is not sufficient, as by default BOSS tries to autodetect the game it is installed for, and if there is no game present it will exit with an error. In this circumstance, the following instructions should be used to obtain masterlist updates:

  1. Copy BOSS GUI.exe from your BOSS folder to the computer with an Internet connection.
  2. Run the BOSS GUI.exe and select the Update Masterlist Only option. For the Game option below, select your game from the drop-down list. This will tell BOSS which game's masterlist to download.
  3. If you use a proxy to connect to the Internet, also record your proxy settings in the relevant part of the GUI's Settings window.
  4. Select Run BOSS. Your masterlist will be downloaded into the same folder as your BOSS GUI.exe. Some other BOSS files will also be created, you can safely delete these.

A rarer circumstance is when the user connects to the Internet using a proxy. This is most common in enterprise environments, and such users will generally know if they use a proxy. If you use a proxy to connect to the Internet, you must either record your proxy settings in the relevant part of the GUI's Settings window or manually enter your proxy settings into your BOSS.ini settings file. See the Customising How BOSS Works section above for details on the expected settings. BOSS does not support the use of authenticated proxies at this time.

If your circumstances are such that the computer you have BOSS installed on is not connected to the Internet, or if you use a proxy to connect to the Internet, it is recommended that you avoid using the auto-updater, simply because it is easier in these circumstances to download new releases through your web browser.

There may also be other special cases which cause problems for BOSS's updaters that the BOSS team is unaware of. If you are having issues using the updaters, contact the BOSS team for support.


Running BOSS Under Linux

It is possible to run BOSS on Linux through the Wine compatibility layer, though no official support for doing so is given. Installation and usage is as on Windows. It is also possible to compile BOSS natively for Linux, and makefiles are provided, although there is again no official support for doing so. Native Linux builds are case-sensitive in their recognition of plugins in the masterlist, and while plugins are generally added with the correct case, occasionally a plugin might have the incorrect case which would cause problems for a user with a native Linux build.

Troubleshooting

Although BOSS is simple to use, it has grown to become a relatively complex application. As such, there are some circumstances under which BOSS may not be able to run correctly, due to a variety of possible reasons. In such cases, an error message detailing the problem should be provided. If the program unexpectedly crashes, or you experience a problem for which no error message is given, please report the crash or problem to an official BOSS thread (linked to in the Quick Links section below), so that we may fix it.

If the information below does not help you to solve any problems you are experiencing, please seek further support in one of the official BOSS threads.

General Problems

There are a few general reasons why BOSS may fail to function correctly. Check if any of the symptoms listed below fit your issue.

  • Symptom: No change in load order.
    • Issue: Wrye *ash's Lock Times load order locking feature is enabled.
    • Solution: Disable Lock Times before running BOSS. Wrye Bash v277 or later and corresponding versions of Wrye Flash and Wrye Flash NV do this for you, and Lock Times is then re-activated after BOSS has run.
  • Symptom: BOSS does not change/update the BOSS Log and/or modlists correctly.
    • Issue: The BOSS Log and/or modlists are read-only, or the BOSS folder is read-only.
    • Solution: Ensure that the BOSS folder and any files in it are not read-only.
  • Symptom: BOSS does not recognise the plugins in my userlist correctly, and/or replaces some of the letters in their names with question marks.
    • Issue: Your userlist is not encoded in UTF-8.
    • Solution: Save your userlist in the UTF-8 encoding.
    • Issue: You had plugins in your userlist with non-Latin letters in their names, and saved your userlist in a non-UTF8 encoding before saving it in UTF-8.
    • Solution: Rewrite the filenames in your userlist, copy/pasting them from the files themselves to ensure they are correct.
  • Symptom: Some plugins have the wrong or no version number given in the BOSS Log.
    • Issue: This occurs because BOSS looks for the version number in the plugin's description field. If the plugin author has forgotten to update the version number there or not provided one, an incorrect or no version number will be displayed.
    • Solution: No solution, other than the plugin author including a correct version number in the plugin's description field for a later release.

File Permissions

If you are running Windows Vista or Windows 7, BOSS (and other modding programs) may be prevented from working correctly by the UAC security feature. There are four common workarounds to this problem:

  1. Install the game outside the Program Files (or Program Files (x86) if on 64-bit Windows). UAC prevents unauthorised edits to these folders, so by installing the game outside of them, you remove it from UAC's reach, allowing you to use mods more easily.
  2. Deactivate UAC. This can be done from the Control Panel, and will turn UAC off across the whole of your computer. It is up to you whether you feel that you have adequate security measures in place to do this without risk.
  3. Give yourself Full Control permissions over your game install folder. This will allow you to make any edits you desire while keeping UAC active and on guard for any edits made by programs you do not run, and also means that you do not need to reinstall your game to a new location.
  4. Run BOSS as an Administrator. Right-click the BOSS.exe and BOSS GUI.exe, and select Run as administrator.

For those that wish to take the third option and give themselves Full Control permissions, here is a guide:

  1. Right-click the folder you wish to change the permissions for, and select Properties.
  2. In Properties, select the Security tab, and click the Edit... button. A UAC prompt may appear, simply allow yourself to continue.
  3. In the Group or user names box, select the Users option.
  4. In the lower box, check the box opposite Full Control in the Allow column. Press OK to exit. If a UAC prompt appears, allow the change.
  5. In the Properties window, select OK to exit. You should now have Full Control permissions over your chosen folder.

International Character Support

BOSS provides support for sorting plugins containing characters from any language via the Unicode character encoding, specifically UTF-8. All BOSS's files are encoded in UTF-8 by default. However, if you created your userlist.txt before BOSS automatically created one, you will probably need to convert it to the UTF-8 encoding, as it is not Notepad's default encoding. Below are some instructions for converting a file's encoding to UTF-8:

  • Using Notepad:
    1. Open the file in Notepad.
    2. Open the File menu, then select Save As....
    3. In the Save As dialog window that opens, there is a drop-down list labelled Encoding:. Select UTF-8 from this list. Do not select any of the other Unicode encodings, they will not work.
    4. Click the Save button, answering Yes to any confirmation prompts.
  • Using Notepad++:
    1. Open the file in Notepad++.
    2. Open the Encoding menu, then select Convert to UTF-8.
    3. Save the file as normal.

A file only needs to be converted to the correct encoding once, thereafter it will continue to save in the correct encoding whenever you edit it.

Version 1.9.1 - 2011

  • Fixed: Mods in the last group of a masterlist being unrecognised when one or more user rules were enabled.
  • Changed: Copyright license is now GNU General Public License v3.0 for everything but the documentation, and GNU Free Documentation License v1.3 for the documentation.
  • Changed: Updated version of libcurl used to 7.23.1 from 7.22.0.

Version 1.9 - 13 November 2011

  • Changed: Major refactoring of BOSS code into a common library, a Command Line Interface and a Graphical User Interface.
  • Common Library:
    • Removed: Checking for changes to the recognised mod list since last run for the plain text BOSS Log, as this did not work correctly and is not feasible to fix.
    • Removed: Command line parameter -T [ --proxy-type ] [ type ] and BOSS.ini setting sProxyType Internet proxy type specification options. Proxy type is now automatically determined.
    • Fixed: A bug where if a plugin filename started with "SET" or "GLOBAL" (or case variations) then BOSS would incorrectly classify it as a variable or global message line and so result in syntax errors.
    • Fixed: User Rules which failed to add a plugin into the beginning or end of a group still incremented the number of recognised plugins as if the rule application had succeeded.
    • Fixed: Userlist rules which reference plugins that are not installed now have warning messages reported instead of error messages.
    • Fixed: Masterlist conditions now detect ghosted mods correctly.
    • Fixed: The online Fallout: New Vegas masterlist's revision being checked instead of the online Fallout 3 masterlist's revision when updating the Fallout 3 masterlist.
    • Fixed: The Hide Messageless Mods BOSS Log filter not acting when another filter resulted in all a mod's messages being hidden.
    • Fixed: The last group of mods in a masterlist being unrecognised.
    • Changed: More robust error handling.
    • Changed: User rules can now only use up to one sort line and/or one REPLACE message line per rule, and cannot sort mods/groups relative to themselves.
    • Changed: Updated version of libcurl used when building BOSS.exe and BOSS GUI.exe to 7.22.0 from 7.21.7.
    • Changed: Content and slight formatting changes to BOSS Log:

      • The date of the user's masterlist is now displayed in the Summary.
      • Ini parsing errors are now displayed in the General Messages section.
      • The filters are now laid out in a more grid-like fashion.
      • Added whitespace characters between plugin filenames, versions, ghosted labels and CRCs to make copy/pasted text more readable.
      • Coloured links in the dark colour scheme so that they are easier to read.
      • The number of plugins and messages hidden at any one time is displayed below the filters box in italics.
      • The Script Extender And Script Extender Plugins section is now always displayed, with CRCs only being calculated if the Display File CRCs option is set.

      • Removed GUI.Settings and GUI.LastOptions sections.
      • DoStartupUpdateCheck moved to a new BOSS.GeneralSettings section.
      • Removed DisableMasterlistUpdate setting.
      • Added variable type prefixes to variable names (s, i, b).
      • Boolean variables now accept true or false instead of 1 or 0.
      • CommandLineVerbosity renamed to iDebugVerbosity.
      • Debug renamed to bDebugWithSourceRefs.
      • Added bLogDebugOutput setting to BOSS.RunOptions to control the logging of debugging output.
      • Added bUseUserRulesEditor setting to BOSS.GeneralSettings to control how to the GUI chooses to edit user rules.
      • Default values for the BOSSLog.Styles settings are now given, and values are no longer enclosed in {} brackets.
    • Changed: Major rewrite of most of the common code, including more object-orientation of code, removal of some unused functions, implementation of an class for formatted output using generic placeholders and much more.
    • Added: Hide Clean Mods and Hide 'Do Not Clean' Messages BOSS Log filters.
    • Added: Support for TES V: Skyrim and the Skyrim Script Extender.
    • Added: Support for MF2.2 syntax, including C-style comments, regular expression conditions, case-insensitive keywords and expanded conditional shorthands.
    • Added: Built-in debug output logging support.
    • Added: DISABLE keyword to user rules that can be placed before a rule's rule keyword to disable it, instead of commenting it out to do the same.
    • Added: Support for authenticated proxies, using the sProxyUsername and sProxyPassword BOSS.ini settings.
      • Note that the support is untested. Feedback from users with authenticated proxies would be appreciated.
  • Command Line Interface:
    • Changed: The command line parameter --debug, -d is now --debug-with-source, -d.
    • Added: Progress indication for downloads.
    • Added: A new command line parameter log-debug, -l to allow logging of the debug output to the BOSSDebugLog.txt file.
    • Added: BOSS Updater functionality.
  • Graphical User Interface:
    • Added: A User Rules Editor to allow creation and editing of user rules without manually editing the userlist.txt.

    • Changed: Settings window overhauled:

      • Added a Log Debug Output setting to allow logging of the debug output to the BOSSDebugLog.txt file.

    • Changed: Simplified layout of main window options.
    • Changed: Renamed some options to reflect their functions better.
