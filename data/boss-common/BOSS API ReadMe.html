Introduction

This documentation details the BOSS API and is intended for use by third party developers of modding utilities for TES IV: Oblivion, TES V: Skyrim, Nehrim - At Fate's Edge, Fallout 3 and Fallout: New Vegas.

The BOSS API has a C frontend and is available as 32-bit and 64-bit DLLs that can be loaded by other modding utilities so that they can easily take advantage BOSS's functionality and the data stored in its files without having to write their own interfaces and keep such interfaces up-to-date as BOSS evolves.

For those not familiar with BOSS, it is a utility that sets the load order of mod plugins so that incompatibilities are minimised, doing so according to a masterlist that contains the optimal load order and is kept up to date via community submissions to the BOSS Development Team. Each game BOSS supports has a separate masterlist. Users may then choose to override or add to the masterlist's data using user rules via a userlist. BOSS's secondary function is as an information dissemination platform, being able to display messages for recognised plugins in its log, giving compatibility information, requirements, warnings, Bash Tag suggestions for use with the Wrye Bash family of utilities, dirty mod information and more.

Overview

The API currently has the following capabilities:

  • Sort a directory of plugins as if BOSS itself were being run, minus creation of a BOSS Log.
  • Determine the resultant load order of a directory of plugins, as if they were to be sorted by the API, without actually changing their current load order.
  • Download the latest masterlist for any supported game.
  • Extract Bash Tag addition and removal suggestions for plugins from the masterlist and userlist.
  • Extract dirty mod messages from the masterlist and determine cleanliness of plugins.
  • Write a minimal masterlist containing only plugin entries with Bash Tag suggestions and/or dirty mod messages, and the suggestions and messages themselves.

The API can also handle determining version compatibility.

Variable Types

The API uses character strings and integers for information input/output.

  • All API strings are unsigned 8-bit character strings (uint8_t*) encoded in UTF-8.
  • All API numbers and return codes are unsigned 32-bit integers (uint32_t).
  • All API array sizes outputted use the unsigned data type size_t.

The API also introduces two new structures:

typedef struct _boss_db_int * boss_db;

The boss_db structure abstracts the definition of BOSS's internal state while still providing type safety across the API. 

typedef struct {
    uint32_t id;
    const uint8_t * name;
} BashTag;

The BashTag structure gives the Unique Identifier (UID) for each Bash Tag and the corresponding Bash Tag name.

Return Codes

The BOSS API functions uses the following uint32_t return codes to signal how a function completes.

Return CodeDescription
BOSS_API_ERROR_OK The function completed successfully.
BOSS_API_ERROR_FILE_WRITE_FAIL A file could not be written to.
BOSS_API_ERROR_FILE_NOT_UTF8 The specified BOSS file (masterlist or userlist) is not encoded in UTF-8.
BOSS_API_ERROR_FILE_NOT_FOUND The specified file could not be found.
BOSS_API_ERROR_MASTER_TIME_READ_FAIL The modification date of the game's master file could not be read.
BOSS_API_ERROR_FILE_MOD_TIME_WRITE_FAIL The modification date of a plugin could not be set.
BOSS_API_ERROR_PARSE_FAIL There was an error parsing the file.
BOSS_API_ERROR_CONDITION_EVAL_FAIL There was an error evaluating the conditionals and/or regular expressions in the masterlist.
BOSS_API_ERROR_NO_MEM The API was unable to allocate the required memory.
BOSS_API_ERROR_OVERWRITE_FAIL The API was unable to overwrite the minimal masterlist specified.
BOSS_API_ERROR_INVALID_ARGS Invalid arguments were given for the function.
BOSS_API_ERROR_NETWORK_FAIL There was an error in an Internet-related operation.
BOSS_API_ERROR_NO_INTERNET_CONNECTION No Internet connection detected.
BOSS_API_ERROR_NO_UPDATE_NECESSARY Not an error. No update was found for the masterlist.
BOSS_API_ERROR_MAX Matches the value of the highest-numbered return code (BOSS_API_ERROR_NO_UPDATE_NECESSARY). It isn't returned by any functions.

Functions

BOSS API provides the functions listed below to clients. Where a function returns a uint32_t value, this is the function's return code.

Version Functions

bool IsCompatibleVersion (uint32_t bossVersionMajor, uint32_t bossVersionMinor, uint32_t bossVersionPatch);

Returns whether this version of BOSS supports the API from the given BOSS version. Abstracts BOSS API stability policy away from clients.

uint32_t GetVersionString (const uint8_t ** bossVersionStr);

Returns the version string for this version of BOSS. The string exists for the lifetime of the library.

Updater Functions

uint32_t UpdateMasterlist (const uint32_t clientGame, const uint8_t * masterlistPath);

Updates the masterlist at masterlistPath to the latest version of the masterlist for clientGame. If there is no masterlist present, then the latest masterlist will be downloaded to masterlistPath. If the local masterlist is already at the latest version, no masterlist is downloaded. clientGame accepts a game code as given in the previous section.

Lifecycle Management Functions

uint32_t CreateBossDb (boss_db * db);

void DestroyBossDb (boss_db db);

Explicitly manage database lifetime. Allows clients to free memory when they want/need to.

Database Loading Functions

uint32_t Load (boss_db db, const uint8_t * masterlistPath, const uint8_t * userlistPath, const uint8_t * dataPath);

Loads the masterlist and userlist from the paths specified. Can be called multiple times. If an error is encountered, the database remains unchanged. Paths are case-sensitive if the underlying filesystem is case-sensitive. masterlistPath and userlistPath are files. dataPath is a directory. If userlistPath is an empty string, then the userlist is not loaded.

uint32_t EvalConditionals (boss_db db, const uint8_t * dataPath);

Evaluates all conditional lines and regex mods the loaded masterlist. This exists so that Load() doesn't need to be called whenever the mods installed are changed. Evaluation does not take place unless this function is called. Repeated calls re-evaluate the masterlist from scratch each time, ignoring the results of any previous evaluations. Paths are case-sensitive if the underlying filesystem is case-sensitive.

Plugin Sorting Functions

uint32_t SortMods (boss_db db, const uint32_t clientGame);

Sorts the plugins for the game specified by clientGame in the directory specified by dataPath when Load() was called.

uint32_t TrialSortMods (boss_db db, uint8_t ** sortedPlugins, const uint32_t clientGame);

Outputs an array listing the plugins in the directory specified by dataPath when Load() was called, with plugins in the order that they would be placed into if SortMods() were to be called.

Database Access Functions

uint32_t GetBashTagMap (boss_db db, BashTag ** tagMap, size_t * numTags);

Returns an array of the Bash Tags encountered when loading the masterlist and userlist, and the number of tags in the returned array. The array and its contents are static and valid until the db is destroyed or until the Load() function is called. It should not be freed by the client. This function must be called prior to calling GetModBashTags(), to ensure that the Bash Tag map exists so that GetModBashTags() may return the Tags found using the correct UIDs.

uint32_t GetModBashTags (boss_db db, const uint8_t * modName, uint32_t ** tagIds_added, size_t * numTags_added, uint32_t **tagIds_removed, size_t *numTags_removed, bool * userlistModified);

Returns arrays of Bash Tag UIDs for Bash Tags suggested for addition and removal by BOSS's masterlist and userlist, and the number of tags in each array. The returned arrays are valid until GetModBashTags() is called again, the db is destroyed or the Load() function is called. The arrays should not be freed by the client. modName is case-insensitive. If no Tags are found for an array, the array pointer (*tagIds) will be NULL. The userlistModified bool is true if the userlist contains Bash Tag suggestion message additions.

uint32_t GetDirtyMessage (boss_db db, const uint8_t * modName, const uint8_t ** message, uint32_t * needsCleaning);

Returns the message associated with a dirty mod and whether the mod needs cleaning. If a mod has no dirty message, *message will be NULL. modName is case-insensitive. The return values for needsCleaning are:

needsCleaning ValueDescription
BOSS_API_CLEAN_NOThe plugin does not need cleaning or should not be cleaned.
BOSS_API_CLEAN_YESThe plugin needs cleaning.
BOSS_API_CLEAN_UNKNOWNBOSS does not know whether the plugin needs cleaning or not, or if it should not be cleaned.

The message string is valid until GetDirtyMessage() is called again, the db is destroyed or the Load() function is called. The string should not be freed by the client.

uint32_t DumpMinimal (boss_db db, const uint8_t * outputFile, const bool overwrite);

Writes a minimal masterlist that only contains mods that have Bash Tag suggestions and/or dirty messages, plus the Tag suggestions and/or dirty messages themselves. outputFile is the path to use for output. If outputFile already exists, it will only be overwritten if overwrite is true.

Contact

The BOSS Development Team can be contacted by Private Messaging any of the handles given in the Project Members section of the main BOSS readme at the noted forums.

However, for API-specific support, it is best to contact WrinklyNinja (available on both the Bethesda Softworks Forums and The Nexus Forums), who is currently responsible for all BOSS code development, including the API. Other team members are may not be of great assistance with coding-related queries.

Developers can also request additional functionality, report any bugs encountered and discuss anything API-related in the offical BOSS API thread.
